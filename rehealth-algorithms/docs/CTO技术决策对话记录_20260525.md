## 二、与 Gemini 的架构讨论（2026-05-25 上午）

### 讨论要点

1. **商业逻辑闭环确认**: Predict → Intervene → Attribute → Settle (PIAS)
2. **16个临床维度定义**: 血压、血糖、血脂、BMI、运动频率等
3. **归因引擎设计**: PSM（倾向得分匹配）+ Bootstrap 重采样 + HMAC 防篡改
4. **"换SIM卡"逻辑**: 归因引擎配置驱动，跨病种零代码改动

### Gemini 方案演进（4个阶段）

**阶段 1 - 初始方案**: 全端侧
- Base 模型 (Gemma 4 E4B) + 端侧 LoRA + 端侧 PSM + 端侧 HMAC
- 完全对应初代 PHM 文档

**阶段 2 - 引入 DRE**: 提升归因准确率到 85%
- 双重稳健估计（Doubly Robust Estimation）替代普通 PSM
- 加入季节/周末时序混杂变量
- 7天洗脱期（Washout Period）隔离

**阶段 3 - 归因移至云端**: 解决前端作弊与算力问题
- 关键论证：归因和签名在手机端 → 越狱可篡改 → 保司不认
- 端侧脱敏 + 云端 DRE + 云端 HMAC 签名
- 云端 FastAPI 微服务架构

**阶段 4 - 大模型也移至云端**: 工程可行性纠偏
- 最终方案：端侧只保留"高密度记忆文件 + 轻量特征提取"
- 大模型推理和归因全部在云端
- 记忆文件实现"越用越懂你"（取代端侧 LoRA）
- App 体积从 3GB 降到 50MB，兼容 8GB 机型

### Gemini 最终架构结论

```
端侧: 健康记忆文件 (JSON, ~5KB) + 特征提取算子 + 做梦机制 (统计聚合)
云端: 大模型推理 + DRE归因 + HMAC签名
```

## 三、Claude (Opus) 的独立技术评估

### 认同的部分

1. **端云分离大方向正确** - 原始数据不出端 + 云端做归因签名，信任模型对保司有说服力
2. **记忆文件替代端侧 LoRA** - 工程最实际的路径，降低风险一个数量级
3. **"换SIM卡"配置驱动** - 跨病种复用的抽象层设计合理
4. **做梦机制** - 夜间静默更新，用户无感知的个性化积累

### 提出的四个修改建议

#### 修改 1: 风险评分用 CatBoost（已训练好），不用 LLM

**问题**: 16 维是强结构化数值数据，LLM 处理数值时存在"数值幻觉"（把 "145" 当 token 序列处理，不理解数值大小）。保险精算师需要可解释性（SHAP 特征贡献），LLM 做不到。

**方案**: 分两层
- **风险评分**: CatBoost (`rehealth_v2_final.pkl`, AUC=0.847, NHANES 2015-2016) → 精确数值 Y + SHAP 贡献度。**模型已基于 NHANES 数据训练完成并验证，exercise_days 是最大可干预因子（重要性排名#2）。**
- **自然语言**: LLM API（Claude/GPT）只负责把评分结果翻译成用户读得懂的干预处方

**老板可能会问**: "不用大模型做评分，那我们的大模型有什么用？"
**回答**: 大模型做它擅长的事——自然语言生成、个性化对话、健康陪伴（AInudge 场景）。评分这种数值任务，精算师只认可解释的统计模型。两者分工，而不是让 LLM 包打天下。

#### 修改 2: 端侧不做 LoRA 训练，只做记忆文件

**问题**: 手机上做反向传播 → 发热降频/OOM/灾难性遗忘。单人数据太稀疏，本地微调容易过拟合。

**方案**: "越用越懂你"通过记忆文件的统计量积累实现，上传到云端后作为 RAG 上下文注入 LLM。效果一致，工程复杂度降一个数量级。

**老板可能会问**: "那我们的专利（端侧 LoRA）不是白写了？"
**回答**: 专利布局保持不变，这是技术壁垒和长期竞争优势。V1 用记忆文件快速落地拿结果，V2 在硬件成熟后（12GB+ 机型普及）逐步兑现端侧 LoRA。两步走，不冲突。

#### 修改 3: 单人归因样本量不足，需要分级策略

**问题**: 一个人 30 天数据，扣掉洗脱期可能只剩 20 个有效点。Bootstrap 置信区间会很宽，精算师不一定认。

**方案**: 双轨归因——面向用户的"预测"和面向保司的"归因"分开做
- **Level 1 个人预测** (面向用户): 用单用户的风险分时序做趋势外推，在 App 里展示"维持现状 vs 执行计划"两条走势线。这是产品卖点，让用户看到"这个 App 对我有用"。
- **Level 2 群体归因** (面向保司): 按脱敏基线特征（年龄段、BMI 分级、血压分级、运动水平）做跨用户 PSM 匹配，比较干预组 vs 对照组的健康指标变化 → DRE + Bootstrap → 群体级控费效果证据。

**为什么群体归因比单人归因更好？** 单人只有 50-60 个数据点，置信区间太宽；群体归因 100+ 用户匹配后，N 足够大，CI 窄到精算师可以签字。而且群体归因天然利用了我们已经在做的脱敏上传——不需要额外数据采集。

**老板可能会问**: "保司什么时候才会真正掏钱？"
**回答**: Phase 1（M1-M6）Level 1 个人预测先跑起来，用户 App 里看到风险走势，留存率上去，同时数据池在积累。Phase 2（M6+）有 100 人 × 90 天数据后启动 Level 2 群体归因，出统计显著的控费报告。预计 M8-M10 可以正式对接保司结算。

#### 修改 4: HMAC 换成 Ed25519 非对称签名

**问题**: HMAC 是对称密钥——保司拿到密钥后也能伪造报告，没有不可抵赖性。

**方案**: Ed25519（非对称）: 云端私钥签名，保司公钥验签。谁也无法伪造，法律效力等同电子签章。

**老板可能会问**: "改动大吗？"
**回答**: Python 几行代码的事，`cryptography` 库原生支持，零额外成本。

## 四、最终架构决策

### V1 工程实施版（当前开发）

```
端侧:
  ┌─ 数据采集 (HealthKit / 华为健康 / Google Fit)
  ├─ 16维特征提取算子 (纯CPU, mean/std/slope/分级)
  ├─ 本地健康记忆文件 (user_health_memory.json, AES加密)
  ├─ 夜间做梦机制 (统计聚合, 10-30秒, 不启动GPU)
  └─ 脱敏打包上传 (不可逆, 无隐私)

云端:
  ┌─ CatBoost 风险评分 (已训练 AUC=0.847, SHAP可解释)
  ├─ LLM API 干预处方 (Claude/GPT, 后期可迁移自建)
  ├─ 双轨归因引擎:
  │   ├─ Level 1 个人预测 (时序趋势外推, 面向用户App)
  │   └─ Level 2 群体归因 (PSM匹配 + DRE, 面向保司结算)
  └─ Ed25519 非对称签名 (HSM私钥, 不可抵赖)
```

### V2 完整愿景版（Future Work）

```
V1 基础上叠加:
  ┌─ 端侧 Gemma 4 E4B (LiteRT, 12GB+机型)
  ├─ 端侧 LoRA 增量学习 + EWC 防遗忘
  ├─ 联邦学习 (跨用户模型进化)
  ├─ 群体级 Meta-Analysis 归因
  ├─ Claude Agent 矩阵 (MCP 协议)
  └─ 区块链锚定 + 零知识证明
```

### 关键保障

- V2 是 V1 之上的叠加，不是替代
- 端侧大模型失败时，自动降级到 V1 记忆文件路径
- 专利权利要求在 V2 中完整落地

## 六、面对老板的关键话术

### "为什么不完全按初代文档做？"

> "初代方案的技术愿景完全正确，专利和竞争壁垒的价值不变。但种子轮阶段最重要的是 6-8 个月内跑出真实数据向保司证明价值。V1 用最低风险的路径跑通全链路、拿到种子用户数据，V2 在数据和硬件成熟后逐步兑现端侧大模型。这是'先活下来再变强'的工程节奏。"

### "用户看到的体验有区别吗？"

> "完全没有。V1 和 V2 对用户来说体验一致——App 每天越来越懂他，干预建议越来越精准。区别只在底层实现：V1 靠记忆文件 + 云端 RAG，V2 靠端侧 LoRA。用户不关心技术栈，只关心建议准不准。"

### "专利不就白写了吗？"

> "没有。专利保护的是端侧 LoRA + EWC + 做梦机制的完整技术路径，这在 V2 中会完整落地。V1 阶段的记忆文件方案也受专利保护（权利要求9：脱敏元数据上报）。而且竞争对手如果想抄我们的最终形态（端侧大模型），会踩到我们的专利。"

### "为什么不自己训练大模型，用别人的API？"

> "CatBoost 风险评分和 DRE 归因引擎是我们自己写的核心壁垒——这两个才是保司付钱的理由。LLM 只是'翻译层'，负责把评分结果变成用户读得懂的话。就像我们用 AWS 服务器不代表我们不是技术公司一样。而且现在没有自有医疗对话数据可以微调，Claude/GPT 开箱即用效果就很好。等用户量起来、积累了自己的交互数据，再考虑自建模型——那时有真实数据来微调，效果比现在凭空训练好得多。100 用户每天调一次 API 的成本不到 100 块/月，自建一台 GPU 服务器要 8000/月。"

### "保险公司什么时候开始掏钱？"

> "分两步。Phase 1（M1-M6）个体级归因给用户看趋势报告，同时积累种子数据。Phase 2（M6+）有 100 人 × 90 天数据后，出群体级 Meta-Analysis 控费报告，这是保司精算师签字的依据。预计 M8-M10 可以开始正式对接保司结算。"
