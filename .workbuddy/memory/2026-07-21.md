## 后端重建拓扑澄清（答疑）

用户问：Win11 本机已有 Docker Desktop，后端为何要去 WSL2 里 docker compose？核查结论：
- **概念上**：Docker Desktop（Win11）底层就是 WSL2，向 Windows shell 和各 WSL distro 暴露**同一个** daemon。`docker` 从哪边调都行，无 Docker 层面的强制要求。
- **实际上**：`deploy/staging/` 编排目录（compose.yml、.env、Dockerfile、mysql/）**只存在于 WSL** `~/rehealthAI/backend/deploy/staging/`；Windows `D:\rehealthAI\backend\` 有 `jeecg-boot/` 源码但**完全没有 deploy/ 目录**。compose 必须在 compose.yml 所在地跑 → 所以从 WSL 跑。
- **源码修复在 WSL 也在**：WSL 的 `jeecg-boot` 源码里 `CvdFeatureVectorDto.java` 有 6 个 `@JSONField`（Windows 7 个），从 WSL 重建能带上修复。✓
- **探针必须从 Windows 跑**：`outputs/probe_mobile_fixed.py` 纯 stdlib，目标 `127.0.0.1:8080`；WSL2 默认 NAT 下 `127.0.0.1` 够不到 Windows 宿主机发布的 8080。正确分工：**compose 重建在 WSL，探针在 Windows**。
- 已把该拓扑事实写入 `MEMORY.md`「Backend deploy topology」节。

## 后端重建 + 探针终验（规则 B 闭环）

操作路径：WSL Maven 编译 → Docker 镜像重建 → Windows 探针验证。全程在本机 Docker Desktop 在线下完成。

### 步骤
1. **准备环境**：Docker Desktop 已启动；WSL 安装 Maven（`sudo apt install maven`，JDK 17 就绪）。
2. **Maven 构建 JAR**：`cd ~/rehealthAI/backend/jeecg-boot && mvn clean package -DskipTests` → BUILD SUCCESS，生成 `jeecg-system-start-3.9.2.jar`（309MB），包含 rehealth 模块的 `@JSONField` 修复。
3. **修复 model-service 端口**：Windows 保留端口 7961-8060 → 8000 被占 → `.env` 改为 `MODEL_PORT=8090, MODEL_BIND_ADDRESS=0.0.0.0`。
4. **Docker rebuild**：`cd ~/rehealthAI/backend/deploy/staging && docker compose up -d --build backend` → Image Built（COPY 层 NOT CACHED，新 JAR 写入），4 容器全 healthy（mysql/redis/model-service/backend）。
5. **探针验证**：`python outputs/probe_mobile_fixed.py`（Windows 侧，目标 127.0.0.1:8080）
